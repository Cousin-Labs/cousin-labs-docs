3 · Producer Workflow — Step by Step
Human + AI
0 · Insights Review
Claude reads the spine and surfaces winning cells, fatiguing formats, unexplored coverage gaps. Producer reviews and applies context (creator availability, brand moment, team capacity). Informs what gets briefed.

Human + AI
1 · Brief Creation
Producer defines cell (creator × format × scenario). Claude generates brief skeleton from library coverage + probability map. Brief ID created here: first record in Concepts DB.

Producer selects creator from roster. Claude surfaces scorecard: prior performance on this cell, variance tier, win rate. Human makes the call.

Creator delivers asset. Claude suggests Brief → Creative match. Human confirms. c1 ID assigned. Brief ID and CL ID are married at this step, not before.

Ad goes live in Meta. Spend begins. Launch record created in spine, tied to the Creative via CL ID. This is the moment the bet becomes testable.

6 · Performance In
Results flow in. Claude pulls SP Score, P1/P2, ITR, spend, CPM into the Paid Testing Tracker, matched to the CL ID and Concept record.

Claude scores vs win_definition, drafts bet verdict (confirmed / refuted / inconclusive), logs to Debrief Log. Adjudication collapses Launches → Creatives. Producer reviews before it feeds back to the strategy generator and next month's briefs.

6 · Notion DB Build — Next Steps
IRL workshop prep with Wade: aligning on DB structure before building. Wade owns usability.
1
Build the Concepts DB
The missing link between Creator Roster and Paid Testing Tracker. Brief ID gets created here first, before a CL ID exists. Fields to define: Concept Name, Brief ID, CL ID, Cell (format × scenario), Persona, Hook Type, Claude Brief Draft. Relations to wire: → Creator Roster, → Brand, → Paid Testing Tracker.
Notion buildNew DB
2
Lock the Brief ID → CL ID handoff
When does the Brief ID in Concepts DB get married to the CL ID? Right now it's informal: the IDs exist in separate places with no defined moment of connection. Needs a named step in the workflow and a named owner. Wade and producer to align on this at the workshop before the DB goes live.
Workflow designHuman gate
3
Wire Debrief Log → Decision Log
Standing decisions that come out of a debrief (retire a format, double down on a cell) currently live nowhere queryable. Add a relation from Debrief Log to the existing Decision Log DB so those calls are captured once and stay visible across future briefs.
Notion buildRelation
4
Build Ladder filtered views across existing DBs
Creator Roster, Paid Testing Tracker, Brand Engagement, and Debrief Log all need a Ladder-scoped view. Same databases, new client filter, not a new system. Once Ladder personas and testing structure are defined, the views follow in an afternoon. This is the first real proof of the thin adapter working.
Notion buildLadder
5
Build a usability guide in Notion
One internal page that answers: what tool do you use for what? Creator Bot for quick lookups. Claude Code + repo for analysis and documents. Notion for tracking and outputs. Right now producers figure this out by fumbling. A single baked read, consistent across views and linked from the hub, removes that friction without handing the system map to clients.
Notion buildInternal
